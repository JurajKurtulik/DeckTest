The Challenge

Ingestion can be harder than it looks.

Every customer has dozens of sources, multiple formats, and conflicting cadences. Getting it right is the foundation everything else stands on.

The Challenge

Three dimensions of complexity compound fast.

Variety100+

Source system types in a typical mid-market estate: SQL, NoSQL, files, APIs, SaaS apps, IoT, mainframe.

Velocityms -> days

Some data must arrive in seconds. Some can wait until midnight. One pipeline cannot optimize for both.

VolumeMB -> TB

A pilot may be a few hundred MB. Production can grow into terabytes, so the cost model has to scale linearly.

The mistake: picking one ingestion tool, then forcing every scenario into it. Fabric gives you five. Use the right one each time.

The Evolution

ETL -> ELT -> Zero-ETL

Fabric supports all three eras. The skill is knowing when each fits the customer reality.

1990s-2010s

ETL

Extract, Transform, Load. Curated and governed, but slow, brittle, and expensive to change.

2010s-today

ELT

Extract, Load, Transform. Flexible, fast, and scalable, but still creates another copy.

2023+

Zero-ETL

Read in place. Zero copy and near-real-time, with stronger dependency on open formats and governance.

The Five Patterns

Five ways to bring data into Microsoft Fabric.

All five land data in OneLake. All five are part of Fabric: one bill, one governance model.

Decision Framework

Which pattern when?

Use Mirroring

Near-real-time operational database replication.

Use Mirroring when the source is supported and the customer needs low-latency changes in OneLake without scheduled CDC pipelines.

Rule of thumb: prefer Shortcuts and Mirroring when possible. They save storage, money, and copy bugs.

Pattern Deep Dives

Pipelines: the workhorse.

When you need control flow, orchestration, and 200+ connectors, this is the natural next step from Azure Data Factory.

Best for data engineers

Use for multi-step jobs, on-prem ingestion, lift-and-shift from Azure Data Factory, parameters, triggers, and managed connectors.

200+ managed connectors Control flow: If/else, ForEach, Until, Lookup, Switch Triggers, schedules, managed identity, gateway, VNet integration

Zero-ETL Options

Do not move data unless the scenario needs it.

Shortcuts

A virtual pointer from OneLake to data that lives elsewhere. Reads happen on the source. No copy. No sync job. No drift.

  • ADLS Gen2
  • Amazon S3
  • Google Cloud Storage
  • Dataverse
  • Another OneLake

Mirroring

Continuous replication of an OLTP source into Delta tables in OneLake. No ETL code and no Fabric CU on the replicate side.

  • Azure SQL Database
  • Azure SQL MI
  • SQL Server on-prem
  • Cosmos DB
  • Snowflake
  • Databricks Unity Catalog
  • PostgreSQL / MySQL
  • Oracle

Best Practices

Incremental loads: pull what changed, not everything.

The single biggest CU saving in any Fabric ingestion design.

Full refresh

Truncate-and-reload every run.

  • Simple, no watermark logic
  • Always consistent with source
  • Slow at scale and high CU per run
  • Heavy source load and throttling risk

Incremental

Pull only what changed since last run.

  • Fast: minutes instead of hours
  • Cheap: fraction of the CU
  • Source-friendly small queries
  • Idempotent and safe to re-run

Rule of thumb: if a table is over 1 GB or runs more than once a day, it should be incremental.

Landing Modes

Overwrite, append, or upsert?

Pick the destination mode that matches the data's nature.

Overwrite

Replace the table every run.

Best forSmall lookups, reference tables, snapshots without history.RiskLose history, expensive on big tables.

Append

Add new rows only.

Best forEvents, audit logs, immutable facts, streaming destinations.RiskDuplicates on re-run without de-dupe.

Upsert / Merge

Update existing, insert new by key.

Best forDimensions, master data, idempotent re-runs.RiskNeeds key columns, slowest of the three.

Capacity-Friendly Ingestion

Do not burn your customer's CU.

Five operating habits keep an F-SKU healthy under load.

01

Stagger schedules

Spread heavy loads across the night instead of running everything at 02:00.

02

Right tool, right job

Cost low to high: Shortcuts, Mirroring, Copy Job, Pipelines, Dataflow Gen2.

03

Push filters to source

Use source WHERE clauses instead of pulling rows you will discard.

04

Pause non-prod

Dev and test capacities can pause overnight and weekends.

05

Watch the Metrics App

Per-item CU views spot noisy items before production throttles.

Fabric's 24h smoothing absorbs short bursts, but sustained overuse throttles every workload on the capacity.

The Expensive Lessons

Five mistakes that burn CU.

Mistake

Dataflow Gen2 as Bronze landing for high-volume tables.

Fix

Use Copy Job or Pipelines for raw movement.

Mistake

Full refresh nightly on a multi-GB table.

Fix

Switch to incremental with watermark or CDC.

Mistake

Every ingestion job scheduled at the same hour.

Fix

Stagger across a 2-hour window.

Mistake

Scheduled CDC pipelines for a source Mirroring supports.

Fix

Use Mirroring: free for replicate and lower latency.

Mistake

No source-side filter.

Fix

Push predicates to the source query.

Governance & Cost

What ingestion costs, and what it touches.

Capacity usage

  • Pipelines: per activity plus copy duration
  • Dataflows Gen2: higher, full Power Query engine
  • Copy Job: per copy duration only, lowest
  • Shortcuts: pay only when data is read
  • Mirroring: free for replicate, only storage

Governance & lineage

  • End-to-end lineage in Purview
  • Sensitivity labels travel with data
  • Tenant-level audit logs
  • Shortcuts inherit source ACLs

Wrap Up

Six takeaways for partner conversations.

  1. Fabric gives you five ingestion patterns. Use the right one.
  2. Shortcuts beat copies; Mirroring beats scheduled CDC.
  3. Default to incremental. Full refresh is the big CU mistake.
  4. Pick the load mode that matches the data: overwrite, append, upsert.
  5. Stagger schedules. Smoothing is not a free pass against throttling.
  6. Sub-second streaming? Eventstream. That is Week 5.
